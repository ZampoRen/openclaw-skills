# PostgreSQL 性能优化深度解析：配置参数决定性能上限

> **摘要**：绝大部分程序员对数据库优化的理解只限于 SQL 层面，但真正决定 PostgreSQL 性能上限的是数百个配置参数。本文通过实测数据揭示：合理调参可带来 30%-343% 的性能提升。

---

## 一、为什么你的 SQL 优化失效了？

在日常工作中，程序员常用的数据库优化招数就是 **query + index + explain 的大三角**：

1. 通过 `EXPLAIN ANALYZE` 分析 query 的执行方案
2. 找到那些会出现 sequential scan 的地方
3. 补上相对应的 index 提高扫描速度

这招平时都挺好用的，直到某一天你遇到了一个解释不清的结果：

> **比如我这里明明有个 index，为啥它还是选择了 sequential scan 而不是 index scan 呢？**

这是因为 **Query Planner 的规划算法比你想的要复杂不少**。它的决策不是一个简单的 decision tree，而是会考虑到诸多因素。

比如上面这个 query，它会先估算你最后可以匹配到的数据量。当它发现这个总量很少的时候，就会觉得说如果我先扫 index 再跳到 table 去读取数据，来回跳转的成本说不定比直接扫 table 读取数据还要高。那不如干脆就跳过 index 直接做 sequential scan 了。

**你可以把 Query Planner 看作是 SQL 的 compiler**，就像我在第 28 期视频里面提到过的：**别跟 compiler 耍小聪明，它的心思比你多太多了**。

---

## 二、真正的幕后黑手：配置参数

Explain 只会告诉你一个在真空中的理论值，但在现实环境里你的 query 不是一个独立事件。你的数据库时刻都在处理各种的 query，还有各种的后台任务在执行，大家都需要排队分享各种资源：

- 如何分配 CPU 和内存
- 要开几个并行的线程
- 缓存多少数据
- 什么时候写入硬盘

这些都会在极大程度上影响数据库整体以及每一个 query 的运行性能。

**而掌控这一切的，是 PostgreSQL 底下的数百个配置参数。可以说它们才是决定 PostgreSQL 性能上限的真正幕后黑手。**

---

## 三、关键参数详解

### 3.1 shared_buffers：缓存的平衡艺术

配置参数看似只是一些简单数字，但是要把它们配好并不是简单的事情，因为你不能够无脑地直接拉到最大值，**不是什么东西都是越大越好的，尺寸合适才是最重要的**。

就拿缓存来说，`shared_buffers` 可以说是整个 PostgreSQL 里最重要的参数之一。因为内存比硬盘有量级上的速度优势，所以我们肯定是希望能够尽量多的缓存数据，让查询更多的从内存拿数据而不是从硬盘上拿。

但 PostgreSQL 官方文档却不建议你把 `shared_buffers` 设置到太高，**最好是系统内存的 25%**。这是为什么呢？

因为 PostgreSQL 是把数据以文件形式存在很多个文件夹里的，所以每次读取数据的时候都要经过文件系统，这样就会触发系统内核里的缓存，这也会提供一个强有力的加速。但操作系统上的缓存是动态管理的，如果当前的内存所剩空间不多了，它就会主动舍弃一些缓存，这样就会影响到数据库读文件的性能。

**所以我们要兼顾 PostgreSQL 和系统缓存，找到一个合适的平衡点。这也是从第一期视频开始我就反复提到过的——取舍的艺术。**

### 3.2 max_files_per_process：分区表的性能瓶颈

有些时候 PostgreSQL 的参数甚至会直接受到系统参数的影响。比如 `max_files_per_process`，可以同时打开的文件数量，这个默认是 1000。

一般情况下已经够用了，因为 PostgreSQL 会尽量把同一个表的数据存在同一个文件里，所以一般的查询都不会需要打开多少个文件。但有一个特殊情况是：

> **如果你用到了 partition tables，每一个 partition 都会被分开储存，查询它的时候就会可能需要打开很多个文件，这个参数就会变成性能瓶颈。**

但你也不能够随便改的，因为在操作系统上也有一个类似的参数，PostgreSQL 是无法突破系统的上限的。比如说 Ubuntu 和 Fedora 这些家用的 Linux 系统，默认的单进程就最多只可以同时打开 1024 个文件。

这主要是因为系统要顾全大局，要确保每一个程序的需求，不能够让一个程序吃光所有的资源。不过我们在生产环境里面，通常都会给数据库准备一个独立的服务器，本来就是要把所有资源都给数据库用的，自然就可以大胆地把系统的上限和 PostgreSQL 上限都往上改了。

### 3.3 enable_partition_pruning：V11 之后的福音

上面这个 partition 的性能问题，在 **V11 之后的 PostgreSQL 已经不是问题了**。因为 V11 有一个新参数 `enable_partition_pruning`，能够给 Query Planner 加上一个更智能的 partition 检测逻辑。

比如说你有一个储存日志的表，通过月份做了 partition。当你搜索今年 1 月到 3 月的数据的时候，Planner 就会判断有几个 partition 可以用得上，而那些用不上的 partition，他们所在的文件都不需要被打开，数据也不会被读取，那整个数据库的压力就会小很多。

那如果我的 PostgreSQL 是 V11 之前的版本怎么办？也不用着急，因为有一个已经被淘汰的老参数 `constraint_exclusion` 可以提供到类似的功能，但它的功能只作用于 inherit table 这种过时的老语法。

## 四、400 个参数，从何下手？

这也反映出来这套知识体系的学习难度所在。因为 PostgreSQL 每年都会稳定地发布大版本更新，每年都会有新的参数出现，旧的参数被送走，你就得一直学一直跟上。

**截止到最新的 V18，PostgreSQL 总共有不多不少刚好 400 个配置参数。**

我不建议你一口气把它们全部都啃下来，因为这是不可能的。所以我推荐你从一些重要的基础参数开始。

这里我分享一个很好用的小网站：**PostgresConf**（https://postgresconf.org/）

这是一个西班牙的小公司做的，它们收集整理了所有的配置参数，包括：
- 它们的加入版本、移除版本
- 默认值、可以调整的范围
- 官方的说明文档
- 如何优化的建议

所有的信息都一目了然。这个小网站有一个调参教程，分为基础、进阶、高级三种。基础教程里面列了 **15 个参数**，如果你没什么头绪的话，就可以从这 15 个下手一个个熟悉。

## 五、实测数据：343% 的性能提升

那么问题来了，如果把所有的参数都弄懂了，都优化了，能够达到什么样的性能效果呢？3%、30% 还是 300%？

废话不多说，我们直接上机测试一下。

### 5.1 测试环境

这里我准备了三台配置一样的服务器：
- **配置**：8 核 16G 内存
- **数据库**：PostgreSQL 13
- **并发连接**：20 个（模拟生产环境的 Connection Pool）

| 组别 | 配置说明 |
|------|----------|
| 第一台 | 保留默认配置参数，无任何修改 |
| 第二台 | 手动优化 15 个基础参数 |
| 第三台 | 所有 PostgreSQL 配置参数 + 操作系统参数全优化（华为云自动调优） |

### 5.2 测试工具

框架用的是 PostgreSQL 自带的 **pgbench**，它不是 MySQL 的 mysqlslap 那种垃圾。你就看它文档上这一大堆参数，基本上什么场景都可以模拟。

pgbench 自带了三个脚本：
- **select-only**：模拟纯读取
- **simple-update**：以更新为主
- **tpc-like**：混合了 SELECT、INSERT、UPDATE 等命令（本次测试使用）

最后这个是基于数据库行业最权威的 TPC 测试标准设计出来的，参考价值应该比自己手搓的脚本更好。

### 5.3 测试结果

pgbench 实时反馈的统计报告包括三个指标：
- **TPS**：体现吞吐能力
- **Latency Average**：端对端平均用时
- **Standard Deviation**：用时波动

#### 第一台：默认基准组

```
TPS: 6400
Latency: 3ms
标准差：8ms
```

#### 第二台：15 参数人工优化组

```
TPS: 8300
Latency: 2.4ms
标准差：7ms

对比基准组提升：约 30%
```

效果已经非常显著了。因为你要知道，这个测试脚本里都是一些很简单的 SQL 语句，没有什么 JOIN、partition 之类的复杂语法。你要是只在 SQL 层面上优化，再加 100 个 index 也做不出来 30% 的性能提升。

#### 第三台：全参数自动优化组

```
TPS: 22000
Latency: -
标准差：5ms

对比基准组提升：343%
```

**不是 3%，不是 30%，是 300%！**

一个简单的 SELECT 或者 INSERT，能够获得 3 倍以上的性能提升，是不是很反直觉？事实就是如此，很多时候我们其实就是守在金山，却不知道怎么把这个金子挖出来而已。

## 六、稳定性比速度更重要

除了 TPS，Latency 的标准差同样也很重要。有参与过系统维护工作的观众应该都懂，**最美的线条就是 Dashboard 上面的平滑直线**，那些忽上忽下的心电图，则是最让人心上图停的。

很多这种波动情况，都是资源分配的不合理造成的。比如说缓存太容易满，或者 VACUUM 经常和查询用户撞上之类的。

从基准组的分阶段数据就能看到：
- 平均标准差：8ms
- 好的时候：3ms
- 差的时候：16-17ms，甚至能够到 21ms
- **波动幅度：7 倍**

而在全参数优化的机器组：
- 平均标准差：5ms 左右
- 最好：3ms
- 最差：9ms
- **大多数情况下都在 4-6ms 之间**

**所以配置参数的优化不只是起到提速的作用，它代表着资源的分配更丝滑，所以更少出现 Edge Cases。**

## 七、专家经验 vs 大模型黑箱

在测试过程中，我就一直很好奇机器组的这个华为云实例，是怎么做到给这么多参数做优化的？是不是搞了什么时髦的大模型来后面进行操盘？

所以我动用小小的人脉，得知了它背后的研发过程：

1. **收集参数**：把各个版本的操作系统参数、内核参数、PostgreSQL 参数都收集起来，收了大概有 7000 多个
2. **专家入场**：数据库专家结合以前服务过的客户、调优过的数据库，找到了一些最佳案例，然后把这些定为 Global Minima
3. **优化算法**：用上了经典的优化算法 Basic Optimization，对 7000 个参数进行优化推导，直到落入到这个 Global Minima 为止

这就非常的朴实无华，都是实打实的经验库产物，**没有大模型那种解释不清的黑箱推演，确实很让人安心**。

这也让我想起了上个月采访开源大佬朱冰冰时，冰冰提到了一个云服务行业的潜规则：

> 大家都有东西就没有办法做出差异化，所以真正决定胜负的，都是那些专家攒起来的经验和独特的功能。
