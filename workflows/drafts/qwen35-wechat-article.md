## 一、Qwen 3.5 小模型家族：四兄弟，各有所长

先来看看这四个型号的"人设"：

| 型号 | 定位 | 适用场景 | VRAM 占用 |
|------|------|----------|-----------|
| **0.8B** | 边缘设备/IoT | 传感器数据处理、简单指令 | ~500MB |
| **2B** | 移动端/轻量任务 | 聊天机器人、文本分类 | ~1.5GB |
| **4B** | 轻量级 Agent | 多模态任务、自动化流程 | ~3GB |
| **9B** | 推理与逻辑 | 代码生成、复杂推理 | ~6GB |

### 两个技术亮点值得注意

**1. 原生多模态架构**

以前的多模态模型，大多是"文本模型 + 视觉适配器"的拼凑方案。Qwen 3.5 的 4B 模型直接把视觉和文本 token 放在同一个潜在空间处理——这意味着更低的延迟和更好的跨模态理解。

**2. Scaled RL 强化学习**

9B 模型用了 Scaled RL 技术，官方说法是"缩小与大模型的推理差距"。翻译成人话就是：**小模型也能听懂复杂指令了，还不容易胡说八道。**

## 二、本地部署：三步走，10 分钟搞定

我测试了两种部署方式，下面是详细步骤。

### 方式一：Ollama（推荐新手）

```bash
# 1. 安装 Ollama（如果还没装）
curl -fsSL https://ollama.com/install.sh | sh

# 2. 拉取模型（以 4B 为例）
ollama pull qwen3.5:4b

# 3. 验证运行
ollama run qwen3.5:4b "你好，介绍一下你自己"
```

**优点：** 一键安装，自动量化，内存管理优秀
**缺点：** 自定义参数较少

### 方式二：LM Studio（适合调试）

1. 下载 LM Studio（https://lmstudio.ai）
2. 在 HuggingFace 搜索 `Qwen/Qwen3.5-4B-Instruct-GGUF`
3. 选择合适的量化版本（推荐 Q4_K_M）
4. 下载后直接加载，调整上下文长度和温度参数

**优点：** 可视化界面，参数可调，适合测试不同量化版本
**缺点：** 需要手动下载，占用空间较大

### 显存占用实测（MacBook Pro M3）

| 模型 | 量化版本 | 显存占用 | 推理速度 |
|------|----------|----------|----------|
| 0.8B | Q4_K_M | 480MB | 120 tokens/s |
| 2B | Q4_K_M | 1.4GB | 85 tokens/s |
| 4B | Q4_K_M | 2.8GB | 65 tokens/s |
| 9B | Q4_K_M | 5.6GB | 42 tokens/s |

## 三、OpenClaw 配置：让本地模型"上岗"

部署好模型后，下一步是让 OpenClaw 调用它。

### 1. 配置本地模型端点

在 OpenClaw 的配置文件中添加：

```yaml
# ~/.openclaw/config.yaml
models:
  local:
    provider: ollama
    endpoint: http://localhost:11434
    default_model: qwen3.5:4b
    
    # 可选：为不同任务指定不同模型
    tasks:
      chat: qwen3.5:2b
      code: qwen3.5:9b
      vision: qwen3.5:4b
```

### 2. 切换模型（运行时）

在 OpenClaw 会话中：

```bash
# 查看当前模型
/status

# 切换到本地模型
/model ollama/qwen3.5:4b

# 或者使用模型别名
/model local
```

### 3. 实测体验

**场景一：日常对话（2B 模型）**

```
用户：今天上海天气怎么样？
Qwen3.5-2B：上海今天晴转多云，气温 12-18°C，
           建议穿薄外套。需要我帮你查更详细的预报吗？
```

响应时间：~800ms
评价：日常聊天完全够用，语气自然，没有明显的"机器味"。

**场景二：代码审查（9B 模型）**

```
用户：帮我看看这段 Python 代码有什么问题
[粘贴代码]

Qwen3.5-9B：1. 第 3 行有潜在的除零风险
           2. 建议用 pathlib 替代 os.path
           3. 异常处理不够具体，建议捕获特定异常
           需要我帮你重构吗？
```

响应时间：~2.1s
评价：代码理解能力接近 70B 级别模型，但速度快 3 倍。

**场景三：多模态任务（4B 模型）**

```
用户：[上传图片] 这张图表说明了什么？

Qwen3.5-4B：这是一张折线图，显示了 2024 年 Q1-Q4 
           的销售额变化。Q3 达到峰值后 Q4 有所回落，
           建议关注季节性因素...
```

响应时间：~1.5s
评价：原生多模态的优势明显，不需要额外的视觉编码器。

## 四、成本对比：本地 vs 云端

算一笔账（按每天 1000 次请求计算）：

| 方案 | 月成本 | 延迟 | 隐私 |
|------|--------|------|------|
| 云端 API（Qwen-Max） | ¥800+ | 1-3s | 数据出域 |
| 本地部署（9B） | 电费≈¥50 | 0.5-2s | 完全本地 |
| 本地部署（4B） | 电费≈¥30 | 0.3-1s | 完全本地 |

**结论：** 对于高频使用场景，本地部署 3 个月就能回本。更重要的是——**数据不出域**。

## 五、踩坑记录（帮你避雷）

### 坑 1：量化版本选错

一开始我用了 Q8_0 量化，结果 9B 模型直接吃满 16GB 内存。后来换成 Q4_K_M，显存减半，精度损失几乎感觉不到。

**建议：** 除非你有特殊需求，否则 Q4_K_M 是最佳平衡点。

### 坑 2：上下文长度设置

Qwen 3.5 小模型默认支持 32K 上下文，但本地部署时 Ollama 默认只给 4K。需要在 Modelfile 里显式设置：

```dockerfile
PARAMETER num_ctx 32768
```

### 坑 3：OpenClaw 模型切换缓存

第一次切换本地模型后，OpenClaw 会缓存连接。如果 Ollama 重启了，需要手动清除缓存或重启 OpenClaw Gateway。

## 六、总结：值不值得折腾？

**我的结论：非常值得。**

### 适合本地部署的场景

- ✅ 高频对话/聊天机器人
- ✅ 代码辅助（审查、生成、解释）
- ✅ 文档摘要和信息提取
- ✅ 多模态理解（图表、截图分析）
- ✅ 隐私敏感任务

### 建议继续用云端的场景

- ❌ 超复杂推理（数学证明、法律分析）
- ❌ 需要最新知识的任务（小模型知识截止较早）
- ❌ 极低资源设备（0.8B 以下需求）
