# 微信公众号 - 技术教程工作流

## 使用说明

**触发方式：** 直接告诉我选题方向或关键词

**示例指令：**
- "写一篇关于 React Server Components 的教程"
- "Topic: Docker 多阶段构建优化"
- "想写 TypeScript 高级类型的内容"

---

## 工作流步骤

### ① 选题确认
- 用户提供技术主题/关键词
- 确认目标读者水平（入门/进阶/高级）
- 确认文章长度偏好（短/中/长）

### ② 多源资料搜集（工具增强版）

**必须使用的工具：**
- [ ] `web_search` - 搜索最新技术动态、竞品分析（至少 5 条结果）
- [ ] `web_fetch` - 抓取官方文档、技术博客原文（至少 3 个核心页面）
- [ ] 人工补充 - GitHub Issues、技术社区讨论

**搜集清单：**
- [ ] 官方文档检索（必须抓取原文，不依赖记忆）
- [ ] GitHub 相关仓库/Issues（查看实际使用问题）
- [ ] 技术社区讨论（StackOverflow、Reddit、HackerNews、V2EX）
- [ ] 竞品内容分析（其他公众号/技术博客，至少 3 篇）
- [ ] 版本时效性确认（发布日期、最新版本号、已知问题）
- [ ] HuggingFace/模型仓库验证（如涉及 AI 模型，必须确认资源真实存在）

**工具使用规范：**
```
# 示例：搜索最新信息
web_search(query="Qwen 3.5 GGUF 量化版本", count=8, freshness="pw")

# 示例：抓取官方文档
web_fetch(url="https://huggingface.co/unsloth", extractMode="markdown")
```

**输出要求：**
- 所有引用的 URL 必须实际可访问
- 所有版本号必须来自官方来源
- 所有"推荐"必须有对比依据（至少 2 个备选方案）

### ③ 深度内容生成
- [ ] 技术背景 & 为什么需要
- [ ] 核心原理讲解（配图表说明）
- [ ] 实战代码示例（完整可运行）
- [ ] 常见问题 & 错误排查
- [ ] 最佳实践 & 性能优化
- [ ] 延伸学习资源

### ④ 内容校验（增强版）

**代码验证：**
- [ ] 所有代码示例在本地/沙箱环境中实际运行
- [ ] 依赖版本明确标注并验证兼容性
- [ ] 错误处理完整，无"省略号跳过"
- [ ] 输出结果截图或日志记录

**事实验证：**
- [ ] 每个技术断言都有官方文档或权威来源支撑
- [ ] API 签名、参数、返回值逐一核对
- [ ] 版本号、发布日期、特性支持情况确认
- [ ] 多源交叉验证（≥3 个独立来源）
- [ ] **资源链接验证** - 所有 HuggingFace/GitHub/官方文档链接必须真实存在
- [ ] **工具二次确认** - 对关键信息使用 `web_search` 或 `web_fetch` 再次验证

**逻辑性审查：**
- [ ] 因果关系清晰（"因为 A 所以 B"有依据，不是臆断）
- [ ] 数据一致性（前后数字不矛盾，如显存占用、速度数据）
- [ ] 场景覆盖完整（"推荐 X 场景"同时说明"不推荐 Y 场景"）
- [ ] 无绝对化表述（"一定"、"绝对"、"100%"改为"通常"、"大多数情况"）
- [ ] 无循环论证（不用结论证明前提）

**真实世界对齐检查（新增）：**
- [ ] 所有"推荐"是否有真实用户反馈支撑？
- [ ] 所有"性能数据"是否标注测试环境？
- [ ] 所有"最佳实践"是否有适用边界说明？
- [ ] 是否存在"假设存在但实际不存在"的资源？（如编造的 HuggingFace 路径）
- [ ] 是否有过度承诺？（"10 分钟搞定"是否真的 10 分钟能搞定？）

**去 AI 味审校：**
- [ ] 删除空洞套话（"总的来说"、"值得注意的是"、"综上所述"）
- [ ] 删除过度连接词（"首先/其次/最后"、"一方面/另一方面"）
- [ ] 删除无信息量的过渡句
- [ ] 删除"作为 AI"类自我指涉
- [ ] 删除过度礼貌/谦卑表达（"仅供参考"、"如有错误请指正"）
- [ ] 检查是否有编造的引用、数据、案例
- [ ] 语言风格：像技术人员写博客，不像机器生成

**人工视角注入：**
- [ ] 加入实际踩坑经验（如果有）
- [ ] 加入个人判断和观点（"我推荐 X 而不是 Y，因为..."）
- [ ] 加入场景化描述（"当你遇到 X 情况时..."）
- [ ] 加入不完美但真实的细节（"这个方案有局限，但在 Y 场景下够用"）

### ⑤ 真实世界对齐与逻辑审查（新增步骤）

**本步骤目标：** 在文章发布前，强制与真实世界对齐，发现并修正事实错误、逻辑漏洞。

**5.1 资源存在性验证（必须使用工具）**

```
# 对文中所有外部链接进行验证
- HuggingFace 模型路径 → web_search 确认存在
- GitHub 仓库链接 → web_fetch 确认可访问
- 官方文档引用 → web_fetch 确认内容匹配
- 第三方工具/服务 → web_search 确认仍在维护
```

**检查清单：**
- [ ] 所有 HuggingFace 路径真实存在（不是 `Qwen/xxx-GGUF` 这种假设路径）
- [ ] 所有 GitHub 仓库不是 404
- [ ] 所有官方文档引用与原文一致（不是凭记忆复述）
- [ ] 所有推荐的第三方服务仍在运营（不是已停止维护的项目）

**5.2 数据一致性审查**

```
# 检查文中所有数字数据
- 显存占用：前后是否一致？（如 0.8B 模型不能前面说 500MB 后面说 800MB）
- 性能数据：是否标注测试环境？（M3/Intel、量化版本等）
- 时间信息：发布日期、版本发布时间是否准确？
- 成本估算：计算逻辑是否清晰？（电费、API 调用次数假设）
```

**检查清单：**
- [ ] 所有性能数据标注测试环境（硬件、量化版本、上下文长度）
- [ ] 所有时间信息有来源（"2026 年 3 月初"而非"昨天"）
- [ ] 所有成本估算有计算过程
- [ ] 前后数据无矛盾

**5.3 逻辑链条审查**

```
# 检查因果关系
- "因为 A 所以 B"：A 是否真实导致 B？还是相关性当因果？
- "推荐 X"：是否有对比 Y/Z？还是拍脑袋决定？
- "最佳实践"：是否有适用边界？还是放之四海皆准？
```

**检查清单：**
- [ ] 所有"推荐"有对比依据（至少 2 个备选方案）
- [ ] 所有"最佳"有适用场景说明
- [ ] 所有因果关系有依据（不是"我觉得"）
- [ ] 承认知识边界（"截至 2026 年 3 月"而非绝对表述）

**5.4 工具验证记录（新增输出）**

在文章末尾添加"验证记录"模块：

```markdown
---
## 事实验证记录
- 验证时间：YYYY-MM-DD HH:mm
- 工具使用：
  - web_search × 3（关键词：...）
  - web_fetch × 5（URL：...）
- 修正内容：
  - [修正 1] 原"HuggingFace 路径"→ 修正为"Unsloth 路径"
  - [修正 2] 原"3 月 4 日发布"→ 修正为"3 月初发布"
- 待确认事项：
  - [ ] XXX 需要后续实际部署验证
```

**5.5 发现问题后的处理流程**

```
发现事实错误 → 立即修正 → 记录到"更新日志" → 重新执行步骤④和⑤
```

> **原则：** 宁可承认"不确定"，也不要编造"看起来合理"的信息。

## 质量保证原则

1. **真实性** - 所有代码亲自验证，不编造 API/数据/案例
2. **深度** - 不止于表面用法，讲解原理和权衡
3. **时效性** - 标注技术版本，避免过时内容
4. **实用性** - 读者能直接应用到项目中
5. **可追溯** - 重要信息标注来源
6. **人味** - 有观点、有判断、有不完美的真实细节，拒绝机器味
7. **真实世界对齐** - 所有外部资源用工具验证，不与真实世界脱节
8. **逻辑自洽** - 因果关系清晰，数据前后一致，无循环论证

## 📦 OpenClaw 安装与配置指南（macOS/Windows/国内）

### 一、macOS 安装方式

#### 方式 1：官方一键脚本（最推荐）

打开**终端**（Terminal 或 iTerm2），执行：

```bash
curl -fsSL https://openclaw.ai/install.sh | bash
```

脚本会自动：
1. 检查并安装 Node.js（需要 >= 22）
2. 安装 OpenClaw CLI
3. 启动 onboarding 配置

安装完成后初始化：

```bash
openclaw onboard --install-daemon
```

#### 方式 2：npm 安装（已有 Node.js 环境）

```bash
npm install -g openclaw
openclaw onboard --install-daemon
```

#### macOS 特有功能

- **macOS App**：支持原生菜单栏应用（可选）
- **Apple 集成**：可连接 Apple Notes、Reminders、iMessage
- **快捷指令**：支持 macOS Shortcuts 触发

---

### 二、Windows 安装方式

#### 方式 1：官方 PowerShell 一键脚本（最推荐）

以**管理员身份**运行 PowerShell，执行：

```powershell
iwr -useb https://openclaw.ai/install.ps1 | iex
```

脚本会自动：
1. 检查并安装 Node.js（需要 >= 22）
2. 安装 OpenClaw CLI
3. 启动 onboarding 配置

安装完成后初始化：

```powershell
openclaw onboard
```

#### 方式 2：WSL2 安装（官方更推荐）

```powershell

wsl --install

# 2. 重启后进入 Ubuntu
```

#### 方式 3：npm 安装（已有 Node.js 环境）

### 三、国内模型配置（阿里云百炼）

#### 1. 获取 API Key

1. 访问 [阿里云百炼控制台](https://bailian.console.aliyun.com/)
2. 开通「模型服务」
3. 创建 API Key（建议设置调用限额）

#### 2. 配置 OpenClaw

编辑 `~/.openclaw/config.json` 或在 onboarding 时配置：

```json
{
  "models": {
    "default": "bailian/qwen3.5-plus",
    "bailian": {
      "apiKey": "你的 API Key",
      "endpoint": "dashscope.aliyuncs.com"
    }
  }
}
```

| 模型 | 适用场景 | 备注 |

### 七、连接聊天渠道（国内推荐）

#### 飞书（企业/团队首选）

```bash
# 1. 安装 Feishu 插件
openclaw plugins install @openclaw/feishu

# 2. 访问 https://open.feishu.cn/app 创建应用
# 3. 获取 App ID、App Secret

# 4. 交互式添加渠道
openclaw channels add
# 选择 Feishu，输入 App ID 和 App Secret

# 5. 重启并测试
openclaw gateway restart

# 首次使用可能需要配对
openclaw pairing approve feishu <配对码>
```

**飞书生态优势：** 文档、日历、会议、云盘、审批完整集成

#### QQ（个人用户首选）

```bash
# 1. 安装 QQBot 插件
openclaw plugins install @sliverp/qqbot@latest

# 2. 绑定 QQ 机器人
openclaw channels add --channel qqbot --token "你的 Token"

# 3. 重启服务
openclaw gateway restart
```

## 🛠️ 自动化工具建议

**提升效率的工具链：**

| 环节 | 工具 | 用途 | 优先级 |
|------|------|------|--------|
| **事实核查** | `web_search` | 搜索最新技术动态、验证资源存在性 | **必须** |
| **文档抓取** | `web_fetch` | 抓取官方文档原文，避免记忆错误 | **必须** |
| 代码验证 | Docker / Replit | 沙箱环境运行测试 | 推荐 |
| 截图生成 | Playwright / Puppeteer | 自动化代码输出截图 | 推荐 |
| 封面图生成 | Midjourney / DALL-E 3 / Stable Diffusion | 文生图 | 推荐 |
| 语法检查 | LanguageTool / 秘塔写作猫 | 中文语法校对 | 可选 |
| 排版优化 | Md2All / 135 编辑器 | Markdown 转公众号格式 | 可选 |

**OpenClaw 集成：**
```yaml
# 工作流配置示例
wechat_tutorial:
  # 事实核查（必须）
  fact_check:
    enabled: true
    min_web_search: 3      # 至少使用 web_search 3 次
    min_web_fetch: 2       # 至少使用 web_fetch 2 次
    verify_links: true     # 验证所有外部链接
  
  # 代码验证（推荐）
  code_verify:
    enabled: true
    sandbox: docker
  
  # 内容输出
  output:
    auto_generate_cover: true
    multi_platform_export: [zhihu, juejin, blog]
    version_tracking: true
    fact_check_record: true  # 输出事实验证记录
```

**工具使用示例：**
```
# 验证 HuggingFace 资源是否存在
web_search(query="unsloth Qwen3.5 GGUF HuggingFace", count=5)

# 抓取官方文档确认 API 签名
web_fetch(url="https://huggingface.co/unsloth/Qwen3.5-9B-GGUF", extractMode="markdown")

# 搜索最新竞品分析
web_search(query="Qwen 3.5 vs Llama 3.2 本地部署对比", count=8, freshness="pw")
```
